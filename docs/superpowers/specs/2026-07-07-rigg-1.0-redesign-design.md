# Rigg 1.0 Redesign — Design Specification

**Date:** 2026-07-07
**Status:** Approved design, pending implementation plan
**Target:** Rigg 1.0.0 (phased via 0.18 → 0.19 → 0.20)
**Backward compatibility:** None required. No migration tooling. Existing projects re-`init` and `pull --adopt`.

## 1. Motivation

Rigg (0.17) works, but four structural problems and one deadline force a redesign:

1. **Knowledge-source "magic".** Azure-managed sub-resources (`createdResources`, cascade ordering, recreate-bug workarounds in `managed.rs`) are fragile and opaque. Explicit pipelines have proven better in practice.
2. **No project boundary.** Everything in a repo is one implicit project; partial pushes/pulls leave local and cloud inconsistent with each other.
3. **False-positive drift.** Server-side volatile data (timestamps, etags, computed hashes) stored locally causes "changes" that aren't changes.
4. **Stale API versions.** Rigg pins Search `2025-11-01-preview` (agentic retrieval is now GA in `2026-04-01` with renames and breaking changes) and Foundry `2025-05-15-preview` on the `assistants` API — **which Microsoft retires 2026-08-26**. The Foundry migration is existential.

Additional goals for 1.0: all data-source types, service-to-service identity tooling, CI/CD as a first-class use case, OpenAPI specs for custom Web API skills, complete Foundry configuration surface, and a from-scratch `push` that stands up an entire solution.

## 2. Current-state facts (July 2026)

Researched against Microsoft Learn and azure-rest-api-specs; re-verify exact endpoint casing during implementation.

### Azure AI Search (data plane)
- **Stable: `2026-04-01`** — first GA of agentic retrieval. Resources are `knowledgeBases` and `knowledgeSources` (renamed from `agents` / `targetIndexes` through the 2025 previews). Retrieve input is `intents` (was `messages`). Answer synthesis, query planning, and configurable reasoning effort are **preview-only** (stable is extractive-only). Requires billing opt-in `knowledgeRetrieval=standard` on the service. GenAI Prompt skill dropped 6 properties (hard 400 if sent). Index **aliases are GA**. Synonym maps long GA.
- **Preview: `2026-05-01-preview`** — adds File / Azure SQL / Fabric Data Agent / Fabric Ontology / MCP Server / Work IQ knowledge-source kinds, `corsOptions`, GPT-5 model family, new retrieve params.
- **Data source types:** GA: `azureblob`, `adlsgen2`, `azuretable`, `azuresql`, `cosmosdb` (NoSQL GA; MongoDB/Gremlin preview), `onelake`. Preview: `mysql`, `sharepoint`, `azurefile` (stable ref) / `azurefiles` (preview ref — spelling differs per version; validate per api-version). Cassandra unsupported.
- Since March 2024, GET on indexers/skillsets/data sources never returns secrets.

### Microsoft Foundry
- **Data plane: `api-version=v1` (stable)** at `https://{account}.services.ai.azure.com/api/projects/{project}`; token audience `https://ai.azure.com`. Agents have first-class **versions** (`/agents/{name}/versions`). Terminology: Conversations/Items/Responses/Agent Versions (was Threads/Messages/Runs/Assistants). Preview features gated by the `Foundry-Features` header (e.g. `HostedAgents=V1Preview`).
- **Assistants API retires 2026-08-26; classic agents 2027-03-31.**
- **Control plane (ARM):** model deployments = `Microsoft.CognitiveServices/accounts/deployments` (stable ARM `2026-05-01`); connections via ARM PUT (identity-based auth: `ProjectManagedIdentity`); guardrails = RAI policies (referenced by deployments as `raiPolicyName` and by agents via `rai_config`); org-wide guardrail enforcement is Azure Policy (out of scope).
- **"Memory"** = conversations (runtime state, not config). Standard-setup backing stores (files→Storage, threads→Cosmos, vector stores→Azure AI Search) are provisioning-level.

### Identity / RBAC
- Managed identity recommended everywhere. Search→model: system identity used automatically when `apiKey` and `authIdentity` are both empty; user-assigned requires `authIdentity`. Roles: **Cognitive Services OpenAI User** (model access), **Cognitive Services User** (KB answer synthesis), **Search Index Data Reader** (agent grounding on a KB), Search Service Contributor / Index Data Contributor / Reader (Rigg's own data-plane access). Foundry roles renamed to Foundry User/Owner/Account Owner/Project Manager.

## 3. Decisions (settled with Kristofer, 2026-07-06/07)

## 4. Workspace & project model

### 4.1 Layout

```
my-rag/
├── rigg.yaml                     # workspace config — YAML, hand-authored, comments welcome
├── apis/                         # shared OpenAPI specs — JSON
│   └── doc-enrichment.json
├── projects/
│   ├── support-rag/
│   │   ├── project.yaml          # project manifest — YAML
│   │   ├── search/
│   │   │   ├── data-sources/support-blob.json
│   │   │   ├── indexes/support-docs.json
│   │   │   ├── skillsets/support-enrich.json
│   │   │   ├── indexers/support-blob-indexer.json
│   │   │   ├── synonym-maps/support-syn.json
│   │   │   ├── aliases/support.json
│   │   │   ├── knowledge-sources/support-ks.json
│   │   │   └── knowledge-bases/support-kb.json
│   │   └── foundry/
│   │       ├── agents/support-agent.json
│   │       ├── agents/support-agent.instructions.md
│   │       ├── deployments/gpt-5-mini.json
│   │       ├── connections/search-conn.json
│   │       └── guardrails/default-rai.json
│   └── sales-rag/…
└── .rigg/                        # gitignored: .rigg/<env>/<project>/state.json
```

Format rule: **Azure's schema → JSON; Rigg's schema → YAML.** JSON resource files are lossless mirrors of Azure definitions; YAML config files are never round-tripped through Azure so comments are safe.

### 4.2 Projects

- The **directory is the truth**: files in the project folder are the membership. `project.yaml` holds only metadata/settings (name, description, optional service-connection pin). No manifest/file drift possible.
- **Exclusive ownership:** a resource (kind + name) belongs to exactly one project. `validate`, `push`, and `pull` all error on workspace-wide duplicates.
- Push order is computed from the reference graph (data source ← indexer → index → skillset → knowledge source → knowledge base → agent), never declared.

### 4.3 Environments

Defined in `rigg.yaml`; resource files are environment-agnostic (same names everywhere — environments differ by *where*, not *what*):

```yaml
environments:
  dev:
    default: true
    search: { service: mklabsrch }
    foundry: { account: mklabaifndr, project: proj-default }
  prod:
    search: { service: mklabsrch-prod }
    foundry: { account: mklabaifndr-prod, project: proj-default }
```

- Selection: `--env` > `RIGG_ENV` > default flag.
- An environment may define multiple named connections per service type; a project pins one by name in `project.yaml` when more than one exists.
- API versions default to Rigg's built-ins (Search stable `2026-04-01`; preview `2026-05-01-preview` used only for preview-gated kinds/fields; Foundry `v1`), overridable per service in `rigg.yaml`.

### 4.4 State

`.rigg/<env>/<project>/state.json`: last-sync checksums (normalized-content hashes) per resource. Used only for local conflict classification. Gitignored; CI operates without it (pure live semantic diff).

## 5. Resource model, registry, sync engine

### 5.1 Resource kinds

- **Search:** `data-source`, `index`, `skillset`, `indexer`, `synonym-map`, `alias`, `knowledge-source`, `knowledge-base`. Knowledge sources are thin references to an index (e.g. `searchIndex` kind); the pipeline behind them is always explicit files. `managed.rs`, `createdResources` handling, cascade workarounds: **deleted**.
- **Foundry:** `agent` (data plane v1), `deployment` (ARM), `connection` (ARM), `guardrail` (ARM RAI policy).

Resources remain schema-light passthrough (`serde_json::Value`); Rigg's understanding lives in the registry.

### 5.2 Metadata registry

One declarative table per kind, keyed by API-version channel:

| Field | Drives |
|---|---|
| endpoint path + version channel | client routing (fixes 0.17 bug where stable resources used the preview version) |
| volatile fields | stripped on pull, ignored by diff (etags, `@odata.*`, timestamps, computed hashes) |
| read-only fields | never written to files |
| secret fields | `validate` **rejects** files containing key-based credentials |
| reference extractors | dependency graph → push order, validation, `describe` |
| capability matrix | per-version validity (KS kinds, data-source type strings incl. `azurefile`/`azurefiles` spelling, preview-only fields, `Foundry-Features` gates) |
| sidecar-default fields | which long-text fields extract to `.md` sidecars on pull (agent instructions) |

Updating Rigg for a new Azure API version ≈ editing this registry (which `rigg dev api-check` nags about).

### 5.3 Sync semantics

- **Pull:** GET project resources → normalize via registry → rewrite only files whose semantic content changed. Remote resources owned by no project are reported as *unmanaged* in `status`; `rigg pull --adopt <project>` claims them.
- **Push:** semantic-diff each resource; push only changes, in dependency order. **After each successful push, GET the resource back, normalize, and write that as the local file + baseline** — disk always holds Azure's canonical form of the user's intent, killing false positives permanently.
- **Deletes are explicit:** local file deleted → push reports orphan, requires `--prune` or interactive confirm to delete remotely. Remote deleted → pull asks: remove local or re-create. Never silent.
- **Conflict classification** (via `.rigg` baselines): unchanged / local-ahead / remote-ahead / both-changed = conflict. Interactive: AI-assisted merge (both-sides summary, hunk-by-hunk). Non-interactive: exit code 5 + machine-readable report.
- **Diff:** semantic (order-insensitive, array identity by `name`), formats `text|json|markdown` (markdown for PR comments), `--exit-code` for CI.
- **Step-by-step building:** each `rigg new` emits one file; each `rigg push <project>` applies only what changed. Single-component updates never require rebuilding a KS/KB.
- **From-scratch:** `rigg push <project>` on a fresh service creates everything in dependency order — the complete-solution requirement. Secrets/credentials are never read from files; where needed, Rigg prompts and guides the user to safe storage (Key Vault / identity), by design preferring configurations where no secret exists at all.

## 6. CLI surface

```
rigg init                          # create workspace; ARM discovery → rigg.yaml
rigg new project <name>
rigg new <kind> <name> -p <proj>   [--type <ds-type>] [--describe "…"]
rigg new pipeline <name> -p <proj> # full explicit chain
rigg new api <name>                # OpenAPI spec in apis/
rigg copy <src> <dst>

rigg pull  [<project>|--all] [--adopt <project>] [--watch]
rigg push  [<project>|--all] [--dry-run] [--prune] [--yes]
rigg diff  [<project>|--all] [--exit-code] [--format text|json|markdown]
           [--compare-env <e>] [--only <kind>/<name>]
rigg delete <project> --remote
rigg status [<project>]
rigg describe [<project>]
rigg validate [<project>] [--strict]

rigg env list|show|add|remove|set-default
rigg auth login|status|logout
rigg auth doctor [--fix]

rigg ai …                          # enable/disable/config/status/test/skill (as 0.17)
rigg mcp serve|install
rigg ci init github
rigg dev api-check                 # hidden from main help
rigg completion <shell> | version
```

- Resource-selection flags (`--indexes`, `--index <n>`, …) are **removed**; the project is the unit. `--only` exists solely on `diff` (read-only lens).
- Single-resource delete = delete file + `push --prune`. Whole-project teardown = `delete <project> --remote`.
- Global flags: `--env`, `--output json`, `--yes`, `--non-interactive` (implied when stdout isn't a TTY). Every prompt has a flag equivalent.
- Exit codes: 0 success · 1 error · 2 usage · 3 validation failure · 4 auth/permission · 5 drift or conflict.
- Removed commands: `pull-watch` (→ `pull --watch`), `config` (edit `rigg.yaml` directly), old `agentic-rag` scaffold (→ `new pipeline`).
- MCP server keeps its 9 tools re-parameterized by project; `rigg-guide` skill and slash commands rewritten for the new model.

## 7. Foundry (v1)

- New client: `https://{account}.services.ai.azure.com/api/projects/{project}`, `api-version=v1`, audience `https://ai.azure.com`. Assistants client deleted. Preview gates via `Foundry-Features` header from the registry.
- **`agent`:** local file = desired latest; `push` publishes a new agent version; `pull` fetches latest. Git = fine-grained history; Azure versions = deployment history.
- **`deployment`** (ARM `2026-05-01`): model name/version, SKU, capacity, `raiPolicyName`.
- **`connection`** (ARM): identity-based auth only (`ProjectManagedIdentity`); key-based connection files fail validation.
- **`guardrail`** (ARM RAI policy): referenced by deployments (`raiPolicyName`) and agents (`rai_config`). Azure Policy governance out of scope (documented in skill).
- **Sidecar pattern:** any string field may be `{"$file": "relative/path.md"}` — inlined on push, re-extracted on pull when a sidecar exists. Registry marks default-sidecar fields (agent instructions; usable for long skillset prompts).
- **Cross-service references:** agent files reference knowledge bases by name via `x-rigg-ref` annotations; push resolves environment-specific endpoints (KB MCP retrieve tool URL) and injects them. `validate` checks all references resolve within the workspace. This makes one file set deployable to any environment unchanged.
- **Memory:** conversations are runtime state — not managed. Backing-store setup is provisioning (Bicep/ARM) — documented, and its identity chain verified by `auth doctor`, but not managed. Vector-store data excluded.

## 8. Auth, identity doctor, CI/CD

### 8.1 Rigg → Azure
Existing chain (SP env vars → Azure CLI → managed identity) + **workload identity federation** (GitHub OIDC via `azure/login`; `AZURE_FEDERATED_TOKEN_FILE` honored). Per-audience token cache with refresh-on-401. No client secrets anywhere.

### 8.2 `rigg auth doctor`
Builds the identity graph from workspace files — every service-to-service edge:

| Edge | Derived from | Example roles |
|---|---|---|
| Search → Storage/Cosmos/SQL | data-source files | Storage Blob Data Reader, Cosmos DB data reader |
| Search → Foundry models | skillsets, vectorizers, knowledge bases | Cognitive Services OpenAI User; Cognitive Services User (KB synthesis) |
| Foundry agent → Search | agent KB grounding | Search Index Data Reader |
| → Key Vault | encryption/config references | Key Vault Secrets User |

Verifies identity existence (system enabled / user-assigned attached) and RBAC assignments via ARM. Per-edge ✓/✗ report with reasons. `--fix` creates role assignments / attaches identities when the caller has rights; otherwise prints exact `az` commands. AI-enabled: tailored remediation advice.

**Guidance default: user-assigned MI** for multi-service / recreatable / promoted stacks (one identity, survives teardown, assignable before services exist); system-assigned for single-service simplicity. Decision helper in `init` and `doctor`. Both supported in files; scaffolds identity-first.

### 8.3 CI/CD
- Non-interactive contract: flag for every prompt; stable exit codes; `--output json`; `diff --format markdown` for PR comments.
- `rigg ci init github` scaffolds: PR validation (validate + diff comment), deploy-on-merge (push, OIDC), nightly drift detection (diff `--exit-code` → issue). Azure DevOps templates in docs.
- Precedence: flags > env vars (`RIGG_ENV`, `AZURE_*`) > `rigg.yaml`.

## 9. OpenAPI specs for custom Web API skills

- `rigg new api <name>` scaffolds OpenAPI 3.1 JSON pre-shaped to the WebApiSkill contract (`values[] / recordId / data` envelope) in workspace `apis/` — shared across projects, no duplication.
- Skillset WebApiSkills link via `"x-rigg-api": "<name>"` (all `x-rigg-*` annotations stripped before push).
- `validate`: spec exists; skill URI matches a spec path; skill inputs/outputs consistent with spec schemas.
- `describe`: "APIs to implement" section (spec, consuming skillsets) — the handoff artifact for a human or AI agent to implement the Function. Rigg never builds or deploys the API itself.
- Versioning: `info.version` + git; breaking change = new file (`<name>-v2.json`) so versions coexist during migration.

## 10. AI features (Ailloy)

All gated on Ailloy configuration + `rigg ai enable`; `--no-ai` per call; never active in `--non-interactive` except read-only explanations.

1. **Explanations** of diffs/pulls/pushes (as 0.17).
2. **Conflict-resolution assistant** — interactive only: summarizes both sides, proposes merge, hunk-by-hunk confirmation.
3. **NL scaffolding** — `rigg new <kind> --describe "…"` generates a registry-validated draft.
4. **Doctor/validate advice** — remediation tailored to the actual stack.

External agents remain the primary AI path (skill + MCP); the skill instructs them to prefer their own reasoning over Rigg-native AI.

## 11. API watchdog

- `rigg dev api-check`: lists available api-versions (azure-rest-api-specs GitHub repo: Search data plane, Foundry; plus Learn as cross-check), compares to registry-supported versions, exits non-zero with a report when Azure is ahead.
- Repo-local skill triggers it at session start (via rigg's CLAUDE.md).
- Weekly GitHub Action runs it and opens/updates an issue on drift.

## 12. Samples & testing

- `samples/` is **one workspace with three projects**: `quickstart-blob` (minimal explicit pipeline → KS → KB), `agentic-stack` (skillset + custom API spec + KB + agent/deployment/connection/guardrail), `cosmos-sql-patterns` (Cosmos + Azure SQL with correct change-detection policies). Demonstrates multi-project for free; each has a step-by-step README.
- Testing: per-crate unit tests; registry-driven contract tests (every kind × version: normalize/diff/reference fixtures); mocked-HTTP integration tests in CI; `test-complete-enduser-experience` skill rewritten to run all three samples end-to-end against `mklabsrch` / `mklabaifndr` — the 1.0 release gate.

## 13. Implementation approach & rollout

**Approach: evolve existing crates.** Keep semantic diff (`rigg-diff`), normalization engine, auth chain, MCP server. `rigg-core` gains workspace/project model + registry; `rigg-client` splits into search / foundry-v1 / ARM clients; `managed.rs` and assistants client deleted. Dependency-modernization pass included (notably replacing archived `serde_yaml` with a maintained alternative).

**Phases:**

- **0.18** — core re-architecture: workspace/project model, registry, Search `2026-04-01` + `2026-05-01-preview`, Foundry v1 migration (beats 2026-08-26 retirement), explicit-only KS, new CLI, dependency modernization.
- **0.19** — `auth doctor` (+`--fix`), `rigg ci init`, `dev api-check` + watchdog wiring.
- **0.20** — OpenAPI spec linking/validation, AI features (conflict merge, NL scaffolding, doctor advice).
- **1.0.0** — samples complete, e2e green against live Azure, docs (README, GETTING_STARTED, MCP.md, SKILLS.md) rewritten.

**Out of scope for 1.0:** vector-store data upload; Azure Policy governance; conversation/memory data; provisioning of the services themselves (Bicep/ARM/Terraform territory); building or deploying the custom Web APIs described in `apis/`.
