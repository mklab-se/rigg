# Rigg 0.18 — Core Re-architecture Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rebuild Rigg's core around the workspace/project model, the metadata registry, Search `2026-04-01`/`2026-05-01-preview`, and the Foundry v1 API — deleting knowledge-source "magic" and the assistants client.

**Architecture:** Evolve the existing 4-crate workspace. `rigg-core` gains `workspace.rs` (workspace/project config) and `registry.rs` (declarative per-kind metadata); resources stay passthrough `serde_json::Value`. `rigg-client` routes per-kind via the registry and replaces the assistants client with Foundry v1 + ARM deployments/connections/RAI policies. The `rigg` CLI becomes project-scoped.

**Tech Stack:** Rust 2024, clap 4, tokio, reqwest, serde_json (preserve_order), serde_yaml_ng (replacing archived serde_yaml), similar, rmcp.

**Spec:** `docs/superpowers/specs/2026-07-07-rigg-1.0-redesign-design.md` (sections 4–7 govern this phase).

## Global Constraints

- No backward compatibility with 0.17 layouts; no migration code.
- Resource files: JSON. `rigg.yaml` / `project.yaml`: YAML. Rule: Azure's schema → JSON, Rigg's schema → YAML.
- Default API versions: Search stable `2026-04-01`, Search preview `2026-05-01-preview` (used only for preview-gated kinds/fields), Foundry data plane `v1`, ARM deployments `2026-05-01`.
- **No secrets in files, ever** — registry `secret_fields` rejected by validate; scaffolds identity-first.
- All `x-rigg-*` annotations are Rigg-local: stripped before any PUT/POST body.
- Exit codes: 0 success, 1 error, 2 usage, 3 validation failure, 4 auth/permission, 5 drift/conflict.
- Exclusive ownership: kind+name unique across all projects in a workspace.
- Push canonicalization: after every successful PUT/POST, GET the resource, normalize, write to disk + baseline.
- Azure testing: only `mklabsrch` (Search) and `mklabaifndr`/`proj-default` (Foundry). No new Azure services/accounts; resources *inside* those services are fair game. Prefer `free`/smallest SKUs and delete after tests.
- Commit after every green task; conventional commits.
